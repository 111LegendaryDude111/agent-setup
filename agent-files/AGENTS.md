# AGENTS.md

## Назначение

Этот файл описывает правила работы агента внутри репозитория.

Агент должен работать предсказуемо, не делать лишних изменений, использовать нужные skills при написании кода и фиксировать результат своей работы в проектных файлах.

---

## Порядок чтения перед началом задачи

Перед началом любой задачи агент должен прочитать:

1. `README.md` — общее описание проекта.
2. `CONTEXT.md` — долгосрочный контекст проекта, если файл существует.
3. `AGENT_HANDOFF.md` — текущее состояние и заметки предыдущих агентов, если файл существует.
4. `AGENT_TASK_LOG.md` — подробный лог выполнения задачи, если файл существует..

Если какого-то файла нет, агент должен продолжить работу и зафиксировать это в `AGENT_HANDOFF.md`.

---

## Основные правила работы

* Работать только в рамках поставленной задачи.
* Делать небольшие, изолированные и проверяемые изменения.
* Не менять unrelated files.
* Не переписывать архитектуру без явного требования.
* Не добавлять новые зависимости без необходимости.
* Не удалять существующие тесты без причины.
* Не утверждать, что тесты прошли, если они не запускались.
* Если тесты нельзя запустить, явно написать почему.
* Сохранять публичные API, если задача явно не требует их изменения.
* Предпочитать простую, понятную и production-oriented реализацию.

---

## Правила качества кода

При реализации или изменении кода агент обязан соблюдать следующие правила:

* Не добавлять fallback-поведение по умолчанию. Фолбеки разрешены только там, где они явно требуются задачей, спецификацией, существующим контрактом или корректной обработкой ожидаемого edge case.
* Если fallback был добавлен, агент обязан явно написать об этом после реализации в формате: `Функция <название>: реализован fallback для <причина>`.
* Если fallback не добавлялся, агент должен явно написать: `Fallbacks не добавлялись`.
* Не создавать и не раздувать файлы с кодом больше 200 строк. Если файл приближается к лимиту, нужно декомпозировать код на более мелкие модули.
* Если существующий файл уже больше 200 строк, агент не должен увеличивать его без явной необходимости. Новую логику нужно выносить в отдельный модуль, если это не ломает связанность.
* Код должен следовать принципам чистой архитектуры: доменная логика не должна зависеть от инфраструктуры, UI, транспорта, хранилища или внешних API.
* Агент должен делать корректную декомпозицию кода: разделять ответственность модулей, держать интерфейсы узкими, избегать смешивания слоев и не создавать shallow abstractions.
* Паттерны проектирования применять только там, где они реально упрощают изменение, тестирование или изоляцию кода. Не добавлять паттерн ради паттерна.
* Для новых или измененных функций добавлять краткие комментарии на русском языке с описанием того, что делает функция.
* Комментарии должны объяснять назначение функции, а не пересказывать очевидные строки кода.

## Обязательные skills для задач с кодом

Для любой задачи, где нужно писать, изменять, ревьюить или рефакторить код, агент должен использовать skills:

1. `$karpathy-guidelines`
2. `$caveman`

Агент должен применять эти skills перед внесением изменений в код.

Ожидаемое поведение:

* `$karpathy-guidelines` — использовать для чистой структуры, maintainability, naming, модульности и дисциплины реализации.
* `$caveman` — использовать для простоты, прямолинейности, минимализма и защиты от over-engineering.

Если нужный skill недоступен в `.agents/skills/` или в глобальных skills, агент должен:

Остановиться и сообщить пользователю, что скилла нет и его нужно либо установить либо остановиться от выполнения задачи

## Session Context Compaction Protocol

Перед компактацией контекста сессии агент обязан предупредить пользователя, что контекст близок к переполнению.

Предупреждение должно явно сообщать:

* текущая сессия близка к переполнению контекста;
* компактация может потерять важные детали;
* лучше начать новую сессию и передать в нее актуальный `AGENT_HANDOFF.md`, `CONTEXT.md` и краткое описание текущей задачи.

Агент не должен начинать компактацию контекста молча, если есть возможность предупредить пользователя заранее.

## Domain Subagents

В репозитории могут быть отдельные инструкции для специализированных субагентов под конкретные типы задач.

Общие правила из `docs/AGENTS.md` всегда имеют приоритет как базовый контракт работы. Domain subagent rules дополняют эти правила и не должны их отменять.

Для задач машинного обучения, data science, feature engineering, model evaluation, ML pipelines, RAG, embeddings, anti-fraud, analytics или data quality агент должен дополнительно прочитать:

* `subagents/machine-learning-engineer.md`

Если задача явно относится к ML/data, агент должен использовать эти инструкции перед планированием и реализацией.

Если инструкции domain subagent конфликтуют с `docs/AGENTS.md`, агент должен остановиться, явно описать конфликт пользователю и спросить, какое правило применять.

## Handoff Protocol

Во время и после выполнения задачи агент обязан обновлять `AGENT_HANDOFF.md`.

`AGENT_HANDOFF.md` является основным постоянным handoff-файлом репозитория.

При использовании skill `$handoff` агент обязан:

* сначала обновить `docs/AGENT_HANDOFF.md`;
* считать `docs/AGENT_HANDOFF.md` источником постоянного состояния проекта;
* записывать результат `$handoff` как временный handoff текущей сессии, а не как замену `AGENT_HANDOFF.md`;
* ссылаться на `docs/AGENT_HANDOFF.md` из временного handoff вместо полного дублирования содержимого;
* добавить в результат `$handoff` секцию с рекомендованными skills для следующей сессии.

Новая сессия должна начинать работу с чтения `README.md`, `CONTEXT.md`, `AGENT_HANDOFF.md` и временного handoff, если он был создан через `$handoff`.

В `AGENT_HANDOFF.md` нужно фиксировать:

* что было сделано;
* какие файлы были изменены;
* какие команды запускались;
* какие тесты запускались;
* какие решения были приняты;
* что осталось незавершенным;
* какие есть риски или блокеры;
* что должен сделать следующий агент.

Агент обязан обновить `AGENT_HANDOFF.md` перед завершением задачи.

Рекомендуемый формат:

```md
## Handoff Entry

- Date:
- Agent:
- Task:
- Status: completed | partial | blocked

### Completed

- 

### Files Changed

- 

### Commands Run

- 

### Tests / Checks

- 

### Decisions

- 

### Risks / Blockers

- 

### Next Steps

- 
```

## Task Log Protocol

Если существует файл `AGENT_TASK_LOG.md`, агент должен вести подробный лог выполнения задачи.

`AGENT_TASK_LOG.md` используется для хронологического execution log.

`AGENT_HANDOFF.md` используется для краткого итогового состояния.

В `AGENT_TASK_LOG.md` нужно фиксировать значимые действия:

* какие важные файлы были прочитаны;
* какие файлы были изменены;
* какие команды были запущены;
* какие тесты были запущены;
* какие ошибки возникли;
* какие retry были сделаны;
* какие implementation decisions были приняты;
* какие результаты вернули субагенты, если они использовались.

Рекомендуемый формат:

```md
### Log Entry

- Time:
- Agent:
- Action type: inspect | edit | test | command | decision | error | cleanup
- Action:
- Reason:
- Files touched:
- Commands run:
- Result:
- Follow-up:
```

## Repeated Error Protocol

Если агент получил одну и ту же или похожую ошибку дважды, он должен остановить blind retry.

После второй повторяющейся ошибки агент обязан:

1. Определить вероятную root cause, описать человеку проблему, выдвинуть гипотезу решения, предложить варинты решения проблемы и рекомендованную, на основе результата решения проблемы оформить и записать правило в AGENTS.md, для избежания в будещем похожей ошибки.
2. Записать ошибку в `AGENT_TASK_LOG.md`, если файл существует.
3. Записать blocker или принятое решение в `AGENT_HANDOFF.md`.
4. Изменить подход перед следующей попыткой.

Третья попытка разрешена только после изменения стратегии.

Повторяющимися ошибками считаются:

* одна и та же failing command;
* один и тот же failing test;
* одно и то же место в stack trace;
* один и тот же проблемный файл или модуль;
* одинаковая ошибка patch/application;
* одинаковая ошибка tool call.

## Guardrails

Агент не должен:

* раскрывать secrets, tokens, passwords или private data;
* записывать secrets в логи;
* запускать destructive commands без явной необходимости и только после арува человеком;
* удалять файлы без объяснения причины;
* добавлять тяжелые dependencies для простой задачи;
* silently ignore errors;
* ослаблять тесты только ради прохождения;
* пропускать failing tests без объяснения;
* выдумывать результаты команд;
* выдумывать содержимое файлов;
* скрывать нерешенные проблемы.

Если guardrail был нарушен или почти нарушен, агент должен зафиксировать это в `AGENT_HANDOFF.md`.

## Testing Rules

Перед завершением coding task агент должен запустить релевантные проверки, если это возможно.

Примеры команд:

```bash
pytest
npm test
npm run test
npm run lint
npm run typecheck
uv run pytest
python -m pytest
```

Агент обязан зафиксировать:

* какие команды были запущены;
* прошли они или упали;
* если проверки не запускались — почему.

---

## Documentation Update Rules

После завершения значимой задачи агент должен определить, изменился ли долгосрочный контекст проекта.

Если да, нужно обновить `CONTEXT.md`.

Примеры долгосрочного контекста:

* реализованная фича;
* важное архитектурное решение;
* новое ограничение проекта;
* новый known issue;
* изменение data contract;
* изменение public API;
* изменение процесса деплоя или тестирования.

Не нужно переносить шумные execution details из `AGENT_TASK_LOG.md` в `CONTEXT.md`.

---

## Final Response Rules

В конце задачи агент должен ответить кратко, структурно и простым языком:

1. Что изменено.
2. Какие файлы изменены.
3. Какие тесты или проверки запускались.
4. Какие риски или нерешенные вопросы остались.
5. Какие следующие шаги нужны.

Агент должен явно написать, если что-то не было завершено. Не использовать англицизмы если это не название файла/функции/переменной/таблицы или поля в бд, или цитаты какого либо объекта, котоый участвует в реализации, в остальных случаях писать на русском, протым языком.
