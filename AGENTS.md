# AGENTS.md

## Назначение

Этот файл находится в корне репозитория и задает контракт работы агента.

Агент должен работать предсказуемо, не делать лишних изменений, использовать нужные skills при задачах с кодом и фиксировать только полезное состояние в проектных файлах.

## Порядок чтения перед задачей

Перед началом задачи агент читает:

1. `README.md` — общее описание проекта.
2. `agent-files/CONTEXT.md` — долгосрочный контекст, если файл существует.
3. `agent-files/AGENT_HANDOFF.md` — короткое текущее состояние, если файл существует.
4. `agent-files/AGENT_TASK_LOG.md` — подробный журнал, только если нужен контекст прошлых действий.

Если файла нет, агент продолжает работу. Не нужно записывать сам факт отсутствия файла в handoff.

## Основные правила

* Работать только в рамках поставленной задачи.
* Делать небольшие, изолированные и проверяемые изменения.
* Не менять unrelated files.
* Не переписывать архитектуру без явного требования.
* Не добавлять зависимости без необходимости.
* Не удалять существующие тесты без причины.
* Не утверждать, что проверки прошли, если они не запускались.
* Если проверки нельзя запустить, явно написать почему.
* Сохранять публичные API, если задача явно не требует их изменения.
* Предпочитать простую, понятную и production-oriented реализацию.

## Качество кода

При реализации или изменении кода агент обязан:

* Не добавлять fallback-поведение по умолчанию. Фолбеки разрешены только там, где они явно требуются задачей, спецификацией, существующим контрактом или корректной обработкой ожидаемого edge case.
* Если fallback был добавлен, явно написать после реализации: `Функция <название>: реализован fallback для <причина>`.
* Если fallback не добавлялся, явно написать: `Fallbacks не добавлялись`.
* Не создавать и не раздувать файлы с кодом больше 200 строк. Если файл приближается к лимиту, декомпозировать код на более мелкие модули.
* Если существующий файл уже больше 200 строк, не увеличивать его без явной необходимости.
* Держать доменную логику независимой от инфраструктуры, UI, транспорта, хранилища и внешних API.
* Держать интерфейсы узкими, не смешивать слои и не создавать shallow abstractions.
* Применять паттерны проектирования только там, где они реально упрощают изменение, тестирование или изоляцию кода.
* Для новых или измененных функций добавлять краткие комментарии на русском языке, если назначение функции не очевидно из названия и структуры.

## Обязательные skills для задач с кодом

Для задач, где нужно писать, изменять, ревьюить или рефакторить код, агент должен использовать:

1. `$karpathy-guidelines`
2. `$caveman`

Если нужный skill недоступен в `.agents/skills/` или в глобальных skills, агент должен остановиться и сообщить пользователю, что skill нужно установить или отказаться от выполнения задачи.

## Context Compaction

Перед компактацией контекста агент обязан предупредить пользователя, если есть возможность.

Предупреждение должно сообщать:

* текущая сессия близка к переполнению контекста;
* компактация может потерять важные детали;
* лучше начать новую сессию и передать в нее `agent-files/CONTEXT.md`, `agent-files/AGENT_HANDOFF.md` и краткое описание текущей задачи.

## Domain Subagents

Для задач машинного обучения, data science, feature engineering, model evaluation, ML pipelines, RAG, embeddings, anti-fraud, analytics или data quality агент дополнительно читает:

* `agent-files/subagents/machine-learning-engineer.md`

Domain subagent rules дополняют этот контракт и не должны его отменять. Если правила конфликтуют, агент останавливается, описывает конфликт пользователю и спрашивает, какое правило применять.

## Handoff Protocol

`agent-files/AGENT_HANDOFF.md` — не история проекта, не отчет о работе и не журнал команд.

Это короткий volatile-файл для продолжения незавершенной задачи. Он должен быть почти пустым, когда активной передачи нет.

Правила:

* Файл всегда перезаписывается, а не пополняется новыми секциями.
* Хранится только одно актуальное состояние: последняя активная, частичная или заблокированная задача.
* Если задача завершена и следующему агенту не нужно продолжать работу, handoff нужно очистить до `status: clear`.
* Не записывать в handoff историю выполненных действий, списки измененных файлов, команды, проверки, retry и подробные выводы.
* Подробности выполнения писать в `agent-files/AGENT_TASK_LOG.md`.
* Долгосрочные знания, решения, ограничения и known issues писать в `agent-files/CONTEXT.md`.
* Лимит handoff: максимум 8 строк и 900 символов.
* Если для продолжения нужно больше деталей, handoff должен ссылаться на конкретный файл или секцию, а не дублировать содержимое.

Формат при отсутствии активной передачи:

```md
# Agent Handoff
status: clear
updated: YYYY-MM-DD
note: No active handoff. Durable context is in agent-files/CONTEXT.md.
```

Формат при незавершенной задаче:

```md
# Agent Handoff
status: active | blocked
updated: YYYY-MM-DD
task: <одна строка>
resume: <точное следующее действие>
blocker: none | <одна строка>
refs: <1-3 пути или якоря, без пересказа>
```

При использовании skill `$handoff` нельзя вставлять полный результат skill в `AGENT_HANDOFF.md`. Нужно сжать его до формата выше, а подробности оставить в отдельном временном handoff только если пользователь явно попросил.

## Task Log Protocol

Если существует `agent-files/AGENT_TASK_LOG.md`, агент ведет там подробный хронологический журнал значимых действий.

В task log можно фиксировать:

* важные прочитанные файлы;
* измененные файлы;
* запущенные команды;
* запущенные проверки;
* ошибки и retry;
* implementation decisions;
* результаты subagents.

Task log может расти. Агент читает его только когда это реально нужно для текущей задачи.

Рекомендуемый формат:

```md
### Log Entry

- Time:
- Agent:
- Action type: inspect | edit | test | command | decision | error | cleanup
- Action:
- Reason:
- Files touched:
- Commands run:
- Result:
- Follow-up:
```

## Repeated Error Protocol

Если агент получил одну и ту же или похожую ошибку дважды, он должен остановить blind retry.

После второй повторяющейся ошибки агент обязан:

1. Определить вероятную root cause.
2. Описать проблему пользователю.
3. Предложить варианты решения и рекомендованный вариант.
4. Записать ошибку в `agent-files/AGENT_TASK_LOG.md`, если файл существует.
5. Записать в `agent-files/AGENT_HANDOFF.md` только blocker и следующий шаг, если задача остается незавершенной.
6. При durable-решении обновить этот `AGENTS.md` или `agent-files/CONTEXT.md`.

Третья попытка разрешена только после изменения стратегии.

## Guardrails

Агент не должен:

* раскрывать secrets, tokens, passwords или private data;
* записывать secrets в логи;
* запускать destructive commands без явной необходимости и явного approval от пользователя;
* удалять файлы без объяснения причины;
* добавлять тяжелые dependencies для простой задачи;
* silently ignore errors;
* ослаблять тесты только ради прохождения;
* пропускать failing tests без объяснения;
* выдумывать результаты команд;
* выдумывать содержимое файлов;
* скрывать нерешенные проблемы.

Если guardrail был нарушен или почти нарушен, агент фиксирует детали в `agent-files/AGENT_TASK_LOG.md`. В `agent-files/AGENT_HANDOFF.md` писать только активный blocker, если он влияет на продолжение работы.

## Testing Rules

Перед завершением coding task агент должен запустить релевантные проверки, если это возможно.

Примеры:

```bash
pytest
npm test
npm run test
npm run lint
npm run typecheck
uv run pytest
python -m pytest
```

В финальном ответе агент обязан указать:

* какие проверки запускались;
* прошли они или упали;
* если проверки не запускались — почему.

## Documentation Update Rules

После завершения значимой задачи агент определяет, изменился ли долгосрочный контекст проекта.

Если да, нужно обновить `agent-files/CONTEXT.md`.

Примеры долгосрочного контекста:

* реализованная фича;
* важное архитектурное решение;
* новое ограничение проекта;
* known issue;
* изменение data contract;
* изменение public API;
* изменение процесса деплоя или тестирования.

Шумные execution details из `agent-files/AGENT_TASK_LOG.md` нельзя переносить в `agent-files/CONTEXT.md`.

## Final Response Rules

В конце задачи агент отвечает кратко и простым языком:

1. Что изменено.
2. Какие файлы изменены.
3. Какие проверки запускались.
4. Какие риски или нерешенные вопросы остались.
5. Какие следующие шаги нужны.

Агент должен явно написать, если что-то не завершено. Не использовать англицизмы, если это не название файла, функции, переменной, таблицы, поля в БД или цитата объекта, участвующего в реализации.
